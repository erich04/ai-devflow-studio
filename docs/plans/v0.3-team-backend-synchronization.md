# v0.3 Team Backend Synchronization Plan

## Goal

Turn DevFlow Studio from a local Electron MVP plus fixture-backed web/API shells into a team-backed
platform where shared Runs, projects, members, costs, and manager dashboards come from a real backend.

## Code Facts Checked

- `apps/api/src/server.ts` now delegates `/api/runs`, `/api/team/overview`, `/api/skills`, and
  `/api/mcp` to the team route resolver and repository boundary.
- `apps/api/src/repositories/postgres-team-repository.ts` maps Postgres-style rows into shared
  DevFlow domain types, but a runtime `pg` driver is not installed or wired yet.
- `apps/web/app/page.tsx` now fetches team overview data through `apps/web/app/lib/devflow-api.ts`
  instead of importing manager dashboard fixtures directly.
- `apps/worker/src/index.ts` currently computes token rollups from fixture token usage.
- `packages/shared/src/domain.ts` already has the future-facing data origin shape:
  `DataOrigin = 'seed' | 'local' | 'remote' | 'adapter'`.
- `apps/desktop/src/App.tsx` currently only switches between `seed` and `local`; `remote` and
  `adapter` are reserved but not implemented.
- Electron local execution is already real enough to remain a boundary: local projects, test
  commands, Runs, Artifacts, Events, Test Evidence, theme preference, and MCP UI state are stored in
  SQLite schema v2.

## Product Scope

v0.3 should add team synchronization without adding real LLM orchestration, real MCP process
management, HoneyAI adapter execution, or production Electron packaging.

v0.3 must preserve cross-platform Electron compatibility. Windows 11 is the primary Windows target;
Windows 10 is best-effort. See `docs/adr/0006-cross-platform-electron-compatibility.md`.

## Architecture

- Keep SQLite as the Electron local/offline/private source for local repository state and local
  execution details.
- Add Postgres as the team source of truth for shared state.
- Expose team state through `apps/api`.
- Make `apps/web` fetch data from `apps/api` rather than importing fixtures.
- Let Electron read remote state and upload approved local summaries through explicit sync calls.
- Keep `adapter` as a reserved future data source for HoneyAI or other execution engines.

## Cross-Platform Constraints

- Do not introduce macOS-only assumptions in API, Web, Postgres, Electron sync, or smoke-test code.
- Do not hard-code POSIX paths, `/tmp`, forward-slash separators, or `bash`/`zsh` behavior.
- Use Node/Electron cross-platform APIs for local execution and persistence: `path`, `os.tmpdir()`,
  `spawn` with explicit `cwd`/`env`, and Electron `app.getPath('userData')`.
- Keep command safety checks compatible with Windows command surfaces such as PowerShell and `cmd`.
- Ensure path display and redacted evidence summaries tolerate Windows drive letters and backslashes.
- Plan Windows smoke/CI coverage for preload availability, SQLite persistence, directory selection,
  test command execution, and restart recovery.

## Data Boundaries

| Origin | Owner | Purpose | v0.3 Status |
| --- | --- | --- | --- |
| `seed` | Shared fixtures | Empty-state demo and tests | Keep |
| `local` | Electron SQLite | Local repo config, local execution, local preferences | Keep |
| `remote` | API + Postgres | Team projects, Runs, members, manager dashboards, cost summaries | Implement |
| `adapter` | Future integration | HoneyAI or external execution engine snapshots | Reserve only |

## Non-Goals

- No real LLM or multi-agent orchestration.
- No real MCP process lifecycle management.
- No HoneyAI adapter implementation.
- No Electron signing, notarization, auto-update, or release packaging.
- No Windows installer, code signing, or auto-update packaging.
- No Git Markdown knowledge indexing pipeline.

## Completion Criteria

- `apps/api` no longer serves team overview data directly from shared fixtures.
- `apps/web` no longer imports manager dashboard data directly from shared fixtures.
- Electron can distinguish `seed`, `local`, and `remote` data in the UI state model.
- Local SQLite remains the source for private local execution details.
- Team Postgres/API becomes the source for shared manager dashboards.
- Windows compatibility is preserved for local execution, SQLite `userData`, path handling, and
  smoke-test design.
- `corepack pnpm verify` passes reliably.
