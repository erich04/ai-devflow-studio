# DevFlow Studio v0.8: Policy-Aware Delivery Automation

## Summary

v0.8 turns Gate Enforcement outcomes into human-approved delivery actions. The goal is not to make
DevFlow auto-fix code or bypass Gates. The goal is to produce clear remediation plans from policy
warnings/blocks, inject those plans into Coding Agent context, and let a human reviewer decide when
to retry implementation or collect more evidence.

This milestone builds directly on v0.7.5: Gate Enforcement, Knowledge Review, local Test Evidence,
and the managed Coding Agent runtime already exist.

## Key Changes

- **v0.8.1 Remediation Model**
  - Add shared domain types for remediation plans and remediation candidates.
  - Generate remediation candidates from `GateEnforcementDecision.blockingReasons`,
    `warningReasons`, `requiredActions`, Knowledge Governance Checks, Agent Policy Findings, and
    existing Test Evidence.
  - Keep the model deterministic and test-first. Retrieval hits alone must still not count as
    evidence.

- **v0.8.2 Policy Context In Coding Brief**
  - Extend Coding Brief assembly so Coding Agent runs can include the selected remediation plan,
    relevant policy reasons, Knowledge References, prior Test Evidence, and prior Coding Diff
    summaries.
  - Renderer must continue passing IDs and user instruction only; Electron/shared runtime owns the
    final brief assembly.
  - Mark likely refactor seams for implementation: `coding-runtime`, `coding-runner`, future
    `useCodingRun`, and future `CodingAgentPanel`.

- **v0.8.3 Human-Approved Retry Flow**
  - Add an Inspector/Agents action to start a Retry Attempt from a remediation plan.
  - Retry remains human-approved: no automatic coding retry, no automatic Gate override, and no
    automatic repository mutation outside the managed worktree.
  - Persist retry attempts as auditable workflow events/artifacts and link them back to the policy
    reasons they addressed.

- **v0.8.4 Manager Policy Summary**
  - Extend Web/API summaries with project/run-level counts for warnings, blocks, overrides,
    remediation plans, retry attempts, and remaining evidence gaps.
  - Keep raw local paths, raw logs, prompts, and patches out of team summaries.

## Public Interfaces / Types

- Expected shared types:
  - `RemediationPlan`
  - `RemediationCandidate`
  - `RetryAttempt`
  - `PolicyAwareDeliverySummary`
- Expected runtime surfaces:
  - Electron evaluates/generates local remediation plans from existing local state.
  - API/Web consume redacted remediation and retry summaries.
  - Coding Agent brief generation accepts remediation context only through main/shared assembly.

## Test Plan

- Shared unit tests:
  - deterministic remediation generation for missing Agent Review, missing/failed Test Evidence,
    API contract violation, and hard-block remediation text.
  - no remediation candidate treats retrieval-only references as governance evidence.
- Electron/runtime tests:
  - remediation context appears in Coding Brief only after a human starts a retry.
  - Retry Attempt records link to policy reason IDs and prior evidence.
- React/E2E:
  - blocked Gate shows remediation plan and retry CTA.
  - retry CTA creates a managed Coding Agent run only after user confirmation.
  - manager console shows redacted policy/remediation summaries.
- Smoke:
  - `corepack pnpm verify`
  - `corepack pnpm test:electron-smoke`
  - `corepack pnpm test:postgres-smoke` when API/Postgres summaries change.

## Assumptions

- v0.8 does not include HoneyAI bridge, RAG/vector retrieval, real MCP execution policy, automatic
  no-approval fixes, Electron packaging/signing, or release distribution.
- v0.8 starts with shared remediation modeling before touching Coding Agent UI.
- `useCodingRun` and `CodingAgentPanel` are candidate refactors only when v0.8.2/v0.8.3 changes make
  the current `App.tsx` coding path too costly to maintain.
