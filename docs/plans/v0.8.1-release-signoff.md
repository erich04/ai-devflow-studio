# DevFlow Studio v0.8.1 Release Signoff

## Summary

v0.8.1 is a release signoff milestone for the v0.8 Policy-Aware Delivery feature plus the v0.8 smoke
hardening patch (PR #2) and release-planning PR #3. It does not add new runtime product capability.
It verifies the committed state, aligns package metadata to `0.8.1`, records the release point as a
git tag, keeps stale branch cleanup explicit, and creates the v0.9 planning entry point.

The release is produced in verification-first order for automated gates: local checks and GitHub
Actions pass before the version bump and tag are applied. The final human walkthrough remains a
post-tag acceptance checklist for the user; it is recorded separately and must not be marked passed
until a human has clicked through the app.

## Release Decision

This signoff intentionally treats `v0.8.0` and `v0.8.1` differently:

- `v0.8.0` already exists at `a63cce0` and is preserved as a historical v0.8 feature/user-guide
  tag. Do not move or rewrite it.
- `main` now includes PR #2 (`330a437` through merge commit `43b8a4d`), which hardened the Electron
  remediation smoke path after `v0.8.0`.
- Package metadata was still `0.7.5` during signoff preparation. The release bump is therefore
  `0.7.5 -> 0.8.1`, not because
  v0.8 functionality is missing, but because `v0.8.1` is the first package/tag pair that aligns
  package metadata with the completed v0.8 feature plus smoke-hardening state.
- Runtime product behavior is already implemented before this signoff. This document only gates
  verification, version alignment, tag creation, and local branch cleanup.

PR #3 (`devflow-v0.8.1-release-v0.9-planning`) extends this signoff work with v0.9 planning docs
and smoke hardening discovered while validating the release candidate. Those changes are still
release/signoff work, not new runtime product capability.

the v0.8 guide.

## Naming Note

`v0.8.1` is used as the release/version tag following the v0.7.5 signoff precedent: a portfolio and
product milestone version that does not imply npm publishing. This coexists with
`docs/plans/v0.8-policy-aware-delivery.md`'s internal phase numbering (`v0.8.1 Remediation Model`
through `v0.8.4 Manager Summary`), which are completed milestone scope markers, not release tags.
They are distinguished by context: git tags and package versions are release points; plan-internal
phase numbers are scope markers.

## Environment Notes

- If E2E or smoke tests fail because ports `4310`, `4311`, or `5173` are occupied, clear the existing
  process and rerun. Port noise is an environment failure, not a signoff success.
- First-time Playwright or Electron environments may need browser/runtime installation before smoke
  tests can run.
- Postgres smoke can use Docker or a disposable local Postgres cluster. The signoff run should use a
  temporary local cluster with a dynamic port.
