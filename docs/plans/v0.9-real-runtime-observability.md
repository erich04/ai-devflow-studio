# DevFlow Studio v0.9: Real opencode Runtime + Observability + Demo Readiness

## Summary

v0.9 promotes the real opencode Coding Agent runtime from local signoff quality (v0.6/v0.6.x) to a
demonstrable, observable, and trustworthy runtime. The single theme of the milestone is:

> A reviewer can watch a real opencode Coding Agent Run proceed step by step, see its tool calls,
> permission decisions, diff, and Test Evidence, and trust what happened — without needing to read
> logs or trust a fake engine.

The fake coding engine stays as the default `verify` path. Real-opencode hardening is layered onto
the existing `CodingEngineAdapter` seam without breaking the deterministic verification path.

This milestone does not include Windows Electron smoke, Electron packaging/signing, real MCP process
management, RAG/vector retrieval, multi-agent orchestration, or any HoneyAI bridge. Windows
compatibility remains an architectural constraint (ADR 0006), not a v0.9 deliverable.

This plan is intentionally not a blind continuation of prior external review notes. v0.9.1 must
re-check the real opencode contract against the current local toolchain before v0.9.2 changes any
runtime behavior. If current opencode behavior differs from ADR 0009 or the v0.6 signoff notes, the
ADR/research record is updated first and the implementation scope is adjusted.

## Why Now

ADR 0009's opencode HTTP-vs-ACP transport selection was done in v0.6 against opencode `1.17.5`. The
real runtime reached local signoff (manual macOS run with Volcengine Ark `double/ark-code-latest`,
multi-step `bash -> edit -> bash -> bash` permission relay) but was never pushed to release-grade
demonstrability. v0.8 added policy-aware retry on top of that runtime. The runtime underneath has not
been re-examined since v0.6.

Before hardening the runtime, the contract it depends on must be re-confirmed, because the opencode
version and HTTP/ACP surface may have moved.

## Current Runtime Seams (reference, not new work)

These already exist and v0.9 builds on them:

- `packages/shared/src/coding-agent.ts` — Coding Brief assembly, redaction, diff limits.
- `apps/desktop/electron/coding-engine.ts` — `CodingEngineAdapter` interface
  (`ensure`/`start`/`approvePermission`/`cancel`), fake adapter, env-driven real adapter selection.
- `apps/desktop/electron/opencode-http-adapter.ts` — opencode HTTP surface
  (sessions, messages, permissions, diff).
- `apps/desktop/electron/opencode-process.ts` — managed `opencode serve` lifecycle.
- `apps/desktop/electron/coding-runtime.ts` — worktree, bootstrap evidence, diff capture, test
  evidence.
- `apps/desktop/electron/coding-runner.ts` — Electron-side run orchestration and push IPC.
- `scripts/opencode-smoke.ts` — env-gated real-opencode smoke, skipped by default.

## Phases

### v0.9.1 Runtime Contract Refresh

Re-confirm the opencode contract before changing the runtime.

- Re-run the ADR 0009 HTTP-vs-ACP comparison against the current opencode version. Record whether
  `opencode serve` HTTP is still the right first transport, or whether ACP has become the better fit.
- Pin or document the target opencode version for v0.9 (was `1.17.5` at v0.6 signoff).
- Document the demo provider configuration without secrets. The current likely demo profile is
  Volcengine Ark / OpenAI- or Anthropic-compatible coding endpoint with `ark-code-latest`, but the
  exact base URL, protocol family, model ID, env var names, and proxy assumptions must be verified
  in this phase rather than assumed from old screenshots.
- Document the runtime contract surface explicitly: transport, session lifecycle, permission relay,
  cancel, timeout, diff fetch, and Test Evidence capture boundaries.
- Record whether provider credentials are injected only into the managed runtime process and never
  written to global opencode auth files.
- Decide, per contract, what v0.9.2 actually hardens (so hardening is contract-driven, not ad hoc).
- Output: updated ADR 0009 addendum or a new ADR if transport changes; a refreshed
  `docs/research/` note. No runtime code change in this phase.

Exit criteria:

- A short contract report names the tested opencode version, selected transport, provider profile,
  and no-go risks.
- Permission ask/reply, cancel, timeout/default rejection, diff capture, and process cleanup are
  explicitly marked supported, unsupported, or deferred.
- ADR 0009 is amended if the selected transport or boundary changed.

### v0.9.2 Real opencode Runtime Hardening

Push the real runtime from signoff quality to demonstrable quality, on the contract confirmed in
v0.9.1.

- Long-running streamed traces: surface real opencode progress through the existing `coding-runner`
  push IPC, so the UI sees steps as they happen rather than only on completion.
- Permission relay: ensure multi-step permission (`bash -> edit -> bash -> ...`) is observable and
  approvable through the Desktop Agents UI, with expiry behavior intact.
- Cancel: real `abort` semantics through the opencode HTTP adapter, with worktree and `opencode serve`
  process cleanup on cancel.
- Timeout: bounded run lifetime with a clear terminal state when a run exceeds it, distinct from
  cancel and from a normal finish.
- Worktree cleanup: guarantee the managed worktree and the `opencode serve` process are torn down on
  success, cancel, timeout, and error paths.
- Test Evidence capture after a real run, consistent with the fake-engine evidence shape so the Tests
  view does not need runtime-specific rendering.
- Keep `DEVFLOW_CODING_ENGINE=opencode-http` as the explicit real-engine switch; default `verify`
  stays on the fake engine.

Exit criteria:

- `corepack pnpm verify` remains green with the fake engine.
- `DEVFLOW_RUN_OPENCODE_SMOKE=1 corepack pnpm test:opencode-smoke` proves the selected real
  transport on the documented provider profile.
- Cancel, timeout, and process/worktree cleanup have deterministic tests or smoke assertions, not
  only manual observation.

### v0.9.3 Runtime Observability

Make a real run legible to both a developer and a manager, without exposing raw logs.

- Agents page: render the real event trace — tool calls, permission decisions, diff, Test Evidence,
  and token/cost — from the persisted Agent Event/trace/token data, not from new runtime-specific UI.
- Distinguish real-runtime events from fake-engine events in the trace so a reviewer can tell which
  path produced the evidence they are looking at.
- Keep redaction: raw stdout/stderr/cwd/prompts stay local; only redacted summaries sync to
  API/Web, consistent with the v0.5/v0.6 remote summary boundary.
- The 5-minute demo script (v0.9.4) is the acceptance test for whether observability is actually
  legible. If a step cannot be explained in the demo, the observability for that step is incomplete.

Exit criteria:

- A reviewer can distinguish fake-engine and real-opencode evidence in the Agents view.
- Tool calls, permission decisions, diff capture, bootstrap/test evidence, terminal state, and
  token/cost source are visible without exposing raw prompts, raw logs, cwd, or provider secrets.
- Web/API summaries remain redacted and do not upload patches or raw traces.

### v0.9.4 Demo Readiness

v0.9 closes with a demo, not just tests. The demo is the acceptance standard for the milestone.

- A 5-minute demonstration script that walks a reviewer through: project select -> Run -> Gate
  Enforcement -> Remediation -> Retry Coding -> real opencode permission relay -> diff -> Test
  Evidence -> observable trace.
- User guide update and screenshot reconciliation against the real runtime path.
- Dual-path signoff: `corepack pnpm verify` (fake engine, deterministic) AND
  `corepack pnpm test:opencode-smoke` (real opencode) both green.

The demo script is not deferred to v1.0. It is part of v0.9 because the demo is how DevFlow proves
the runtime is understandable, not merely executable.

## Out of Scope for v0.9

- Windows Electron smoke (remains ADR 0006 future expansion; tracked, not delivered).
- Electron packaging, macOS signing/notarization, Windows installer, auto-update.
- Real MCP process management, permissions audit, tool-call telemetry beyond the coding runtime.
- RAG / vector retrieval.
- Multi-agent orchestration.
- HoneyAI adapter or execution-engine bridge.
- Team accounts, RBAC, deployment.

## Test Plan (indicative, finalized per-phase)

- Shared/Electron unit: contract assertions from v0.9.1; cancel/timeout/cleanup terminal-state tests
  in v0.9.2; trace event shape in v0.9.3.
- Smoke: `corepack pnpm verify` must remain green throughout on the fake engine; real-opencode smoke
  extended in v0.9.2/v0.9.4.
- Release notes: each phase records whether validation used fake engine, real opencode, or both.
- Manual: the v0.9.4 demo script is the human acceptance gate.

## Assumptions

- v0.9.1 produces a contract decision before v0.9.2 changes runtime code. If v0.9.1 changes the
  transport, v0.9.2 scope is revisited against the new contract.
- The fake engine remains the default verification path for every phase; real-opencode is opt-in via
  env, as in v0.6.
- Demo readiness (v0.9.4) is a hard part of v0.9, not deferred to v1.0, because the demo is the
  strongest test of whether the runtime is actually demonstrable.

## Next Milestone (after v0.9)

v1.0 candidate scope, deferred until the real runtime is demonstrable and observable: Electron
packaging/signing/auto-update, real MCP/Skill runtime management, RAG/vector retrieval, team
accounts/RBAC/deployment, and a public portfolio demo page.
