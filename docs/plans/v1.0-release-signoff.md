# DevFlow Studio v1.0.0 Release Signoff

## Summary

v1.0.0 signs off DevFlow Studio as a minimum self-hosted team pilot. The release proves that the
product has moved beyond the local portfolio workstation: a small team can authenticate through Web,
create a project, pair a Desktop client, sync redacted local workflow summaries, and view shared
state through the Web console.

This is not a public SaaS release and does not include distribution packaging, automatic HTTPS,
invitation management, billing, real MCP execution, or multi-Desktop concurrency guarantees.

## Scope Signed Off

- v1.0a Identity Foundation
  - Real `User`, `AuthAccount`, authenticated session, and identity-backed team projection types.
  - Demo session behavior remains explicit and separate from authenticated sessions.
- v1.0b GitHub OAuth + Minimal Team Project
  - First-user organization owner bootstrap.
  - GitHub OAuth start/callback/logout boundary.
  - Owner-only minimal project creation.
- v1.0c Desktop Pairing + Authenticated Sync
  - One-time project pairing codes.
  - Scoped Desktop bearer tokens stored through the local credential boundary.
  - Redacted authenticated run-summary sync without falling back to demo headers.
- v1.0d Self-Hosted Minimum Deployment
  - Docker Compose stack for API, Web, and Postgres.
  - `.env.example`, self-hosted pilot guide, and explicit Docker smoke.
  - GitHub Actions Docker smoke coverage.

## Verification Evidence

Release-candidate local verification passed before creating the `v1.0.0` tag:

- `corepack pnpm typecheck` passed.
- `corepack pnpm test` passed: 48 files / 295 tests.
- `corepack pnpm build` passed.
- `corepack pnpm verify` passed, including typecheck, unit tests, cross-platform checks, browser
  E2E, and Electron smoke.
- `git diff --check` passed.

Docker-dependent verification:

- `corepack pnpm test:docker-smoke` is not part of default `verify`.
- It must pass in GitHub Actions for the release commit because local machines may not have Docker.

Release-only real provider verification:

- Real opencode provider smoke remains outside default `verify` and GitHub CI because it can spend
  provider quota.
- Starting with the next product release after v1.0.0, one real `opencode` smoke against the
  configured Doubao/Volcengine provider profile must pass before the release tag is created.
- Record the result using `docs/plans/release-only-real-opencode-smoke.md`.
- Do not substitute fake-engine evidence for this release-only smoke when the release claims real
  runtime behavior.

GitHub Actions verification required before tagging:

- `Docker smoke`
- `Postgres integration`
- `Windows compatibility`
- `macOS verify`

## Release Procedure

1. Commit the version and signoff documentation changes.
2. Push `main` and wait for the GitHub Actions `Verify` workflow to pass.
3. For future releases, run and record the release-only real opencode provider smoke.
4. Create annotated tag `v1.0.0` on the verified release commit.
5. Push the tag.

## Known Boundaries

- The Docker pilot API container runs `tsx src/server.ts` so workspace TypeScript package boundaries
  resolve reliably. A slimmer prebuilt runtime image is future release-engineering work.
- Real opencode and paid provider calls remain explicit smoke/signoff paths, not default CI.
- The final human walkthrough is still a manual acceptance activity; automated Playwright/Electron
  smoke covers the repeatable path, while Computer Use click automation is not treated as the source
  of truth.
