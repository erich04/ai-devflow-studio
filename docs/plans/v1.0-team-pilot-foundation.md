# DevFlow Studio v1.0: Team Pilot Foundation

## Summary

v1.0 moves DevFlow Studio from a strong local/portfolio workstation to a minimum self-hosted team
pilot. The goal is not to launch a public SaaS or complete every collaboration feature. The goal is
to prove that a small team can authenticate, create a project, pair a Desktop client, sync redacted
local workflow summaries, and view the shared state in Web.

This milestone builds on v0.9's local runtime confidence: fake-engine `verify` remains the default
CI path, while real opencode and paid provider calls remain explicit smoke/signoff paths.

## Current Starting Point

- v0.9 has completed real opencode lifecycle hardening, observability, Tool / Skill Timeline, and
  dual-path signoff discipline.
- API already has early team tables: `organizations`, `users`, `projects`, and `project_members`.
- API already has `TeamSession`, demo fallback helpers, and an `allowDemoFallback` switch.
- `TeamMember` is still a UI/display model, not a real identity or membership source.
- Desktop/Web sync still defaults to demo headers and must not be treated as authenticated team
  connectivity.

## v1.0a: Identity Foundation

Establish the identity model before adding OAuth or Desktop pairing.

- Extend the existing team schema instead of creating a parallel membership system.
- Add or formalize real identity concepts such as `User`, `AuthAccount`, and authenticated session
  state.
- Keep `project_members` as the project-membership truth.
- Keep `TeamMember` as a UI projection derived from identity/membership data.
- Separate `DemoSession` from `AuthenticatedSession`; demo fallback is only for tests, seed data,
  and explicit demo mode.
- Preserve current Web/Desktop rendering by projecting existing UI data from the new identity
  foundation.

Completion criteria:

- API can resolve authenticated sessions without using the demo owner fallback.
- Demo fallback is still available only when explicitly allowed.
- Existing Team Console display continues to render through projection data.
- Tests prove demo and authenticated sessions do not silently cross over.

Out of scope:

- GitHub OAuth UI.
- Desktop pairing.
- Invitations.
- Multi-organization switching.

## v1.0b: GitHub OAuth + Minimal Team Project

Add the smallest Web-authenticated team entry point.

- Implement GitHub OAuth start/callback/logout and session cookie handling.
- Bootstrap rule: in a fresh self-hosted deployment, the first successful GitHub login creates the
  default organization and becomes its owner.
- Allow an owner to create a minimal project with name, slug, description, and repository display
  metadata.
- Stop defaulting Web API calls to `u-erich owner`; demo headers remain test/demo-only.

Completion criteria:

- A new user can log in with GitHub and become the first organization owner.
- The owner can create a project and see it in Team Console.
- Protected API calls reject missing or invalid authenticated sessions.

Out of scope:

- Invitation workflow.
- GitHub App installation.
- Repository permission scanning.
- Billing or SaaS onboarding.

## v1.0c: Desktop Pairing + Authenticated Sync

Connect Electron Desktop to the team project with a scoped token.

- Web generates a one-time project pairing code.
- Desktop exchanges the code for a scoped desktop token bound to user, organization, and project.
- Pairing code defaults: 10-minute TTL, high entropy, server-side hash storage, five failed attempts,
  HTTPS required outside localhost.
- Desktop stores tokens through the credential boundary and never writes plaintext team tokens into
  repo files or logs.
- Desktop sync uses the authenticated token and never falls back to demo headers when token auth
  fails.
- Sync remains redacted: no cwd, raw stdout/stderr, prompt, patch body, provider secret, or full
  repository content.

Completion criteria:

- Web can create a pairing code for a project.
- Desktop can pair to the project and sync one redacted run summary.
- Web can display the synced summary under the project.
- Expired or revoked Desktop tokens produce an explicit reconnect state.

Out of scope:

- Token rotation UI.
- Revoke UI.
- Multi-Desktop concurrency guarantees.
- Full audit dashboard.

## v1.0d: Self-Hosted Minimum Deployment

Make the pilot runnable outside the development shell.

- Add Docker Compose for API, Web, and Postgres.
- Provide `.env.example` and setup instructions for database and GitHub OAuth configuration.
- Add a Docker smoke that brings up the stack, applies migrations, uses a test session or mocked
  OAuth path, creates a project, pairs Desktop, syncs a summary, and verifies Web visibility.

Completion criteria:

- A clean machine can start the minimum stack from docs.
- Migrations run successfully in the containerized Postgres.
- The authenticated Desktop-to-Web summary path is smoke-tested.

Out of scope:

- Kubernetes.
- Automatic HTTPS.
- Installer/signing/auto-update.
- Managed cloud hosting.

## v1.0 Release Criteria

v1.0 is complete when the team pilot foundation works end-to-end:

- GitHub OAuth login works.
- First-user organization owner bootstrap works.
- Minimal project creation works.
- Desktop pairing works.
- Authenticated redacted sync works.
- Web displays the team project state.
- Docker Compose runs the minimum stack.
- Demo mode and real auth are visibly separate.
- Default CI remains deterministic and cost-free.

v1.0 does not require:

- Ten-person concurrent real use.
- Invitations or member administration.
- GitHub PR creation.
- MCP real execution.
- RAG/vector retrieval.
- Public SaaS onboarding.
- Electron packaging or app signing.
- Windows real smoke.

## v1.0.x Follow-Up

After v1.0, harden the pilot with:

- Multi-Desktop concurrency tests with at least three clients.
- Member add/remove flows.
- Desktop token revoke/rotation.
- Audit entries for login, pairing, sync, override, and token revoke.
- Backup/restore and healthcheck guidance.
