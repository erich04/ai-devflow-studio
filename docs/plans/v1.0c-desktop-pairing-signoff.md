# DevFlow Studio v1.0c Signoff: Desktop Pairing + Authenticated Sync

## Summary

v1.0c connects the Electron Desktop client to a real team project without relying on demo session
headers. Web issues a short-lived, copy-once project pairing code. Desktop exchanges that code for a
scoped bearer token, stores only encrypted token material locally, and uses the token for redacted
team sync.

This slice does not add token rotation UI, revoke UI, multi-Desktop concurrency guarantees, or a
full audit dashboard. Those remain v1.0.x hardening work.

## Scope Completed

- Pairing code model and API:
  - Shared `DesktopPairingCode`, `DesktopPairingExchangeResult`, and `DesktopPairingCredential`
    contracts.
  - Postgres schema v6 with `desktop_pairing_codes` and `desktop_tokens`.
  - Pairing codes store server-side hashes instead of plaintext codes or secrets.
  - Pairing codes use a 10-minute TTL and five failed-attempt budget.
  - `POST /api/team/projects/:projectId/pairing-codes` requires lead access to the project.
  - `POST /api/desktop/pairing/exchange` exchanges the copy-once code for a scoped bearer token.
- Authenticated Desktop sync:
  - API server resolves `Authorization: Bearer ...` through the repository.
  - Invalid bearer tokens do not fall back to demo sessions.
  - Desktop remote sync uses bearer-token headers when a paired credential exists.
  - Desktop keeps demo-session sync only for unpaired/local demo mode.
- Desktop credential boundary:
  - SQLite schema v7 stores Desktop pairing metadata separately from encrypted token material.
  - Electron IPC exposes `loadDesktopPairing` and `pairDesktop`.
  - Renderer never receives the plaintext bearer token after pairing succeeds.
- Web and Desktop user entry points:
  - Team Console project cards can create a Desktop pairing code.
  - A same-origin Web route proxies pairing-code creation so the browser does not construct demo
    headers or direct cross-origin API calls.
  - Desktop topbar shows paired/unpaired status and accepts a Web-generated pairing code.
- Postgres smoke coverage:
  - Lead creates a pairing code.
  - Desktop exchanges the code for a bearer token.
  - Bearer-token sync uploads a redacted run summary.
  - Team overview includes the synced run and does not leak the copy-once code or bearer token.

  removed it after completion.
- The Desktop bearer token remains runtime/credential material. It is encrypted at rest locally and
  intentionally excluded from renderer state and team overview payloads.
