# DevFlow Studio v1.2: Runtime Cost UX + Budget Administration

## Summary

v1.2 turns the v1.1 Runtime Cost + Budget Guard into an operator-visible workflow. v1.1 already
introduced budget policies, budget approvals, budget evaluation, and the pre-provider-call guard in
Electron. v1.2 keeps those contracts and adds the missing UX:

- Web Team Console can view and edit a project's runtime budget policy.
- Web Team Console can create explicit runtime budget approvals.
- Desktop Agents view shows budget decisions in the Coding Agent trace.
- Desktop retry can carry a user-entered budget approval id to the runtime guard.

Default verification remains deterministic and cost-free. Real opencode + real provider smoke stays
release-only and explicitly environment-gated.

## Delivered Scope

- Added Web client helpers for:
  - `GET /api/runtime/budget-policy`
  - `PUT /api/runtime/budget-policy`
  - `POST /api/runtime/budget-approvals`
- Added a Web Runtime Budget panel showing:
  - policy enabled/disabled state
  - monthly limit, warning threshold, and current spend
  - existing budget approvals
  - forms for saving a policy and creating an approval
- Extended Desktop coding run input with `runtimeBudgetApprovalId`.
- Extended IPC parsing so the renderer may pass `runtimeBudgetApprovalId` while still rejecting raw
  prompts.
- Extended the Electron runtime budget guard boundary with `approvalId`.
- Preserved the v1.1 safety boundary: Desktop sends an approval id, and the API/runtime boundary
  resolves that id to a complete approval record before calling `evaluateRuntimeBudgetGuard`.
- Added Desktop Agents view budget trace details:
  - status
  - reason
  - projected/current/limit cost
  - approval id when present
  - explicit "Retry with approval" control for `requires_lead_approval`.

## Non-Goals

- No new database schema.
- No new external product API.
- No automatic approval matching.
- No approval request queue or notification workflow.
- No CI call to a paid model provider.
- No raw prompt, cwd, stdout/stderr, patch body, or provider secret upload.

## Release-Only Real Provider Smoke

The default `verify` path must stay fake-engine based. Before a release that claims real-provider
budget behavior, run:

```bash
DEVFLOW_RUN_OPENCODE_SMOKE=1 corepack pnpm test:opencode-smoke
```

Record the result in the release signoff. This smoke may consume Volcengine/Doubao provider tokens
and should not run in default CI.
